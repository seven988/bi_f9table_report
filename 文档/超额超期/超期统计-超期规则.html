📊 超期数据统计-超期规则明细

基于月结账期与自然日顺延的客户超期未收分析
版本 V1.1
更新 2026-07-02
数据库 Doris（兼容 MySQL 语法）

一、概述

本 SQL 用于监控客户的 超期未收金额，核心目标是：识别哪些客户的应收账款已经超过了约定的付款期限（账期），并按逾期天数分段统计金额，帮助业务人员快速定位高危逾期客户。

与普通“自然日顺延”账期不同，本 SQL 支持 月结 场景下的复杂到期日计算：即根据结算方式（月结、两月结、三月结等），确定“目标结算月份”，再结合账期天数（AccDays）计算出最终的到期日。整个计算过程分为 3 层嵌套，层层递进，最终输出按客户汇总的超期未收金额及逾期天数分布。

二、整体逻辑流程

📄 内层子查询 → 🔗 OverdueBase CTE → 📊 最终 SELECT
① 内层子查询
关联成本、明细、虚拟合同，计算每笔收入的未收金额、目标结算月、账期天数
② OverdueBase CTE
根据月结规则计算到期日、逾期天数，并过滤超期且非内部公司的记录
③ 最终 SELECT
按委托人+业务员汇总超期未收总额，并按逾期天数区间（1-30, 31-60, 61-90, 90+）分段统计

三、详细逻辑说明

步骤 1：内层子查询（成本 & 明细 & 虚拟合同关联）

🎯 业务目标
从成本单和成本明细中计算每笔收入的 应收、已收、未收，并关联虚拟账期合同获取 账期天数（AccDays） 和 结算方式（决定延后月数），同时计算出 目标结算月份（用于月结场景的到期日计算）。

📥 数据来源

  • f9_Cost：成本单主表，提供客户、业务日期、业务员、客服等。
  • f9_Costitem：成本明细表（仅取收入 Ioid=1），计算应收、已收。
  • f9_CusContract：虚拟账期合同，提供 AccDays、PayWayCd（结算方式代码）。
  • f9_CusCredit：临时账期调整表（TempType='A'），若有有效记录则覆盖合同默认账期。

⚙️ 核心计算

  • 应收 / 已收：按照 ChgSign（* 或 /）和 ChgRate 将金额折算为本位币，保留两位小数。
  • 目标月份（TargetMonthFirstDay）：
    先取业务日期的当月第一天（MonthFirstDay），然后根据结算方式对应的 延后月数 向后推移。
    例如：业务日期 2026-01-15，结算方式为“月结”（延后 1 个月），则目标月份为 2026-02-01。
  • 账期天数：优先取临时账期（TempAccDay），若为空则取合同 AccDays，若合同 AccDays < 1 则默认 25 天。

📤 输出字段（供上一层使用）

  • Entrust、BusWay、JobNoIn、PayWay、CtrlOffice、Saler、Service
  • BizMonth（业务月份）
  • CostName、WhoName
  • UncollectedAmount（未收金额 = 应收 - 已收）
  • TargetMonthFirstDay（目标结算月的第一天）
  • AccDays（账期天数）

⚠️ 重要过滤：只保留 未收金额 > 0 的记录。

步骤 2：OverdueBase CTE（计算到期日与逾期天数）

🎯 业务目标
基于目标结算月份和账期天数，计算每笔收入的 到期日，并与当前日期比较，得到 逾期天数，同时过滤出已超期的记录。

⚙️ 核心计算逻辑（重点）

  • 到期日 的计算规则：
    • 若 AccDays ≤ 目标月份的总天数，则到期日 = 目标月份第一天 + (AccDays - 1) 天（即当月第 AccDays 天）。
    • 若 AccDays > 目标月份总天数，则到期日 = 目标月份的最后一天（即月末）。
    这实际上模拟了 “月结 + 账期天数” 的逻辑：账期天数表示“从月初算起的第 N 天”，如果超过月末则直接取月末。
  • 逾期天数 = 当前日期 - 到期日（若 > 0 则表示已超期）。
  • 过滤：只保留 逾期天数 > 0 的记录，且排除内部公司（WhoName NOT IN f9_Company）。

📤 输出字段

  • 上述所有维度字段 + UncollectedAmount、OverdueDays

步骤 3：最终 SELECT（汇总与分段）

🎯 业务目标
按 委托人（Entrust） 和 业务员（Saler） 分组，汇总超期未收总额，并按逾期天数区间（1-30、31-60、61-90、90+）分别统计金额，同时输出最大超期天数。

📤 最终输出字段

  • 委托人
  • 业务员
  • 超期未收总额
  • 1-30天(轻度)
  • 31-60天(中度)
  • 61-90天(重度)
  • 90天以上(高危)
  • 最大超期天数

💡 业务价值：快速了解每个客户超期金额的分布，聚焦于逾期最严重的客户。

四、超期日期计算示例（重点）

为了帮助理解月结场景下的到期日计算，我们以 “好客贸易有限公司” 的一笔收入为例，逐步演算整个过程。

4.1 基础数据

字段值
业务日期（BizDate）2026-01-15
结算方式代码（PayWayCd）M（月结，延后 1 个月）
虚拟合同账期（AccDays）25 天
当前日期（current_date）2026-07-02
未收金额50,000 元

4.2 计算步骤

① 计算目标结算月份第一天（TargetMonthFirstDay）

  • 业务日期的当月第一天：2026-01-01
  • 结算方式为“月结”，延后月数 = 1
  • 目标月份 = 2026-01-01 + 1 month = 2026-02-01

② 计算目标月份的总天数

  • 2026年2月共有 28 天（非闰年）

③ 判断 AccDays 是否 ≤ 总天数

  • AccDays = 25，≤ 28，因此到期日 = 目标月份第一天 + (25 - 1) 天 = 2026-02-01 + 24天 = 2026-02-25

④ 计算逾期天数

  • 当前日期 2026-07-02 减去 到期日 2026-02-25 = 127 天（已超期）

⑤ 分段归属

  • 127 天 > 90 天，因此该笔未收金额计入 “90天以上(高危)” 区间。

4.3 特殊情况：AccDays 超过当月总天数

场景：同一客户，另一笔业务日期为 2026-03-15，AccDays = 40，结算方式仍为月结。

  • 目标月份 = 2026-04-01
  • 4月总天数 = 30 天
  • 40 > 30，因此到期日 = 2026-04-30（月末）
  • 若当前日期为 2026-07-02，则逾期天数 = 2026-07-02 减去 2026-04-30 = 63 天（重度）

4.4 对比自然日顺延的差异

如果采用简单的“业务日期 + AccDays”方式（非月结），则第一笔的到期日为 2026-01-15 + 25 = 2026-02-09，逾期天数为 143 天，比月结算法多了 16 天。月结算法实际上给予了客户更长的宽限期（因为从月初开始计算），更符合“月结”的商业惯例。

六、f9_CusCredit 在超期计算中的角色（重要）

📌 核心结论

f9_CusCredit（临时账期调整表）只影响账期天数（AccDays），不参与“目标结算月”的决策。“目标结算月”完全由 结算方式（PayWayCd） 独立决定。

6.1 分工图解

graph LR A[业务日期
BizDate] --> B[当月第一天
MonthFirstDay] B --> C{结算方式
PayWayCd} C -->|M/M2/S/S1| D[延后月数] D --> E[目标结算月
TargetMonthFirstDay] F[f9_CusContract.AccDays] --> G[账期天数 AccDays] H[f9_CusCredit.TempAccDay] -.->|覆盖| G E --> I[到期日计算] G --> I I --> J[逾期天数]

蓝色路径 由结算方式独立决定，紫色路径 由 f9_CusCredit 影响

6.2 对比示例

假设业务日期均为 2026-01-15，结算方式均为 月结（M，延后 1 个月），目标结算月固定为 2026-02-01。

场景AccDays 来源AccDays 值到期日计算结果
合同默认账期f9_CusContract.AccDays252026-02-01 + 24 天2026-02-25
临时账期覆盖f9_CusCredit.TempAccDay4040 > 28（2月天数），取月末2026-02-28
临时账期覆盖f9_CusCredit.TempAccDay102026-02-01 + 9 天2026-02-10

📌 结论：

  • 目标结算月始终是 2026-02-01（因为结算方式没变）。
  • f9_CusCredit 只改变了 AccDays 这个数字，从而改变了到期日在目标月份内的具体位置（月初、月中、月末）。
  • 因此，“目标结算月”由结算方式决定，“到期日”由账期天数决定，两者职责分明。

九、注意事项

  • 月结算法依赖结算方式：如果合同结算方式不是月结系列（如现结），延后月数可能为 0，此时目标月份 = 业务月份，账期从当月开始计算，逻辑依然成立。
  • 临时账期覆盖合同账期：系统优先取 f9_CusCredit 中有效的临时账期（TempType='A'），若无则取合同 AccDays。
    重要 f9_CusCredit 只影响 AccDays，不改变目标结算月。
  • 到期日边界处理：当 AccDays 等于目标月份总天数时，到期日正好是月末（符合规则）。
  • 性能建议：该 SQL 涉及多表关联和子查询，建议在 f9_Costitem 的 CostId、Ioid、redid 及 f9_Cost.BizDate 上建立索引。
说明：本文档基于超期应收监控 SQL 逆向提炼，所用函数（如 months_add、days_add、LAST_DAY）为 Doris 特有，若迁移至其他数据库需做相应适配。
