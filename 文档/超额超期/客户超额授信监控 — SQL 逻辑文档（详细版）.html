📊 客户超额授信监控 — SQL 逻辑文档（详细版）

计算客户未收金额与授信额度的差额，识别超额风险
版本 V1.0
更新 2026-06-30
数据库 MySQL（兼容 SQL Server）

一、概述

本 SQL 用于监控客户的信用使用情况，核心目标是：计算每个客户的应收未收金额（即“未收金额”），并与客户当前的授信额度（永久额度 + 临时额度）进行比较，判断是否“超额”。业务人员可以通过此报表快速发现信用风险客户，及时采取跟催或调整授信措施。

查询逻辑分为 4 个逻辑层（CTE），层层递进：先计算每一笔收入明细的未收金额，再汇总客户总额，然后获取客户授信额度（虚拟合同 + 临时授信），最后合并计算差额与状态。

三、详细逻辑说明

步骤 1：CostItemDetail（计算单笔未收金额）

🎯 业务目标
从成本明细表中提取所有收入记录（Ioid = 1），计算每一笔的未收金额（即“应收 - 已收”），并考虑汇率转换，将外币金额统一折算为本位币。

📥 数据来源

  • f9_Costitem（成本明细表）：记录了每一笔收支的金额、已收金额、汇率符号和汇率值。

⚙️ 计算逻辑

  • 只取收入记录（Ioid = 1）。
  • 未收金额 = (原始金额 - 已收金额)，然后根据 ChgSign 进行汇率折算：
    • 若 ChgSign = '*'，则乘以 ChgRate（汇率）；
    • 若 ChgSign = '/'，则除以 ChgRate；
    • 否则直接取原值（即本币）。
  • 最终结果保留 2 位小数，并命名为 单笔未收本位币。

📤 输出字段

  • CostId：成本单ID，用于后续关联。
  • 单笔未收本位币：该笔收入的未收金额（本位币）。

💡 业务解释
每一笔收入在系统中都有“原始金额”（可能是外币）和“已收金额”（已收回的本位币）。通过汇率折算后，我们得到该笔收入还剩余多少未收回的金额。

步骤 2：ValidTempCredit（获取有效临时授信）

🎯 业务目标
从临时授信表中获取当前有效的临时授信额度（类型为 TempType = 'C'），用于后续叠加到总授信中。

📥 数据来源

  • f9_CusCredit（临时授信表）：存储临时额度或临时账期调整记录。

⚙️ 计算逻辑

  • 筛选条件：
    • TempType = 'C'（授信调整类型）；
    • ApproveStatus = '结束'（已审批通过）；
    • CURDATE() BETWEEN EffDate AND ExpDate（当前日期在有效期内）。
  • 取 TempCredit 作为临时额度，并关联到虚拟合同ID（CusBcId）。

📤 输出字段

  • CusBcId：关联的虚拟合同ID。
  • 临时额度：当前有效的临时授信金额。

💡 业务解释
企业有时会给客户授予临时授信（如促销期间增加额度），这些临时授信有固定的有效期。我们只取当前仍在有效期内的记录。

步骤 3：CustomerTotal（汇总客户总未收金额）

🎯 业务目标
将所有收入明细按客户（Entrust，即委托客户）汇总，计算每个客户的总未收金额。

📥 数据来源

  • f9_Cost（成本单主表）：提供客户名称（Entrust）。
  • CostItemDetail（上一步得到的明细未收金额）。

⚙️ 计算逻辑

  • 内连接 f9_Cost 和 CostItemDetail，按成本单ID关联。
  • 按 Entrust 分组，使用 SUM 汇总未收金额，得到 客户总未收本位币。
  • 只保留客户名称非空且未收金额 > 0 的客户（实际查询中在最终筛选，此处先汇总）。

📤 输出字段

  • 客户名称：委托客户名称。
  • 客户总未收本位币：该客户所有未收金额的总和。

💡 业务解释
一个客户可能有多笔业务，每笔业务都有未收金额。我们需要加总得到该客户的整体应收余额，用于与授信额度比较。

步骤 4：最终输出（计算授信额度、差额与状态）

🎯 业务目标
将客户总未收金额与授信额度（永久额度 + 临时额度）进行比较，计算超额差额，并标记“超额”或“正常”状态，同时补充业务员信息。

📥 数据来源

  • CustomerTotal：客户总未收金额。
  • f9_CusContract：合同表（取 CtClass = '虚拟账期合同' 的 CreditAmount 作为永久授信基础）。
  • ValidTempCredit：临时授信额度。
  • f9_Customer：客户基础信息表（获取业务员 Owner）。

⚙️ 计算逻辑

  • 授信额度 = COALESCE(虚拟合同CreditAmount, 0) + COALESCE(临时额度, 0)。
  • 超额差额 = 客户总未收本位币 - 授信额度，若 >0 则表示超额。
  • 授信状态：若 客户总未收本位币 > 授信额度 则为“超额”，否则为“正常”。
  • 关联客户表：通过子查询按 CusName 分组去重，取 MAX(Owner) 作为业务员，确保一个客户只对应一个业务员，避免因多行导致重复。
  • 过滤：排除内部公司（f9_Company 表中存在的客户），只显示外部客户。

📤 最终输出字段

  • 客户名称：委托客户名称。
  • 业务员：该客户的负责业务员。
  • 客户总未收本位币：客户当前所有未收金额之和。
  • 客户授信额度：永久授信 + 临时授信（若无则默认为0）。
  • 超额差额：未收金额 - 授信额度（正数表示超额）。
  • 授信状态：超额 / 正常。

💡 业务解释
通过比较未收金额和授信额度，我们可以快速知道哪些客户已经超出了信用限额，需要重点催收或调整授信政策。超额差额越大，风险越高。

四、涉及的数据表说明（简单版）

为了让不熟悉数据库结构的读者也能理解，下面用通俗语言解释每张表的作用。

表名通俗说明关键字段
f9_Cost成本单主表，记录了每笔业务的基本信息，包括客户（委托方）、业务员等。Id（成本单号）、Entrust（委托客户）
f9_Costitem成本明细表，记录了每笔业务的具体收支项目，如收入、费用，以及金额、已收金额、汇率等。CostId（关联成本单）、Ioid（收支标识）、Amount（金额）、Ysje（已收金额）、ChgSign（汇率符号）、ChgRate（汇率）
f9_CusContract合同表，存储所有合同信息，其中“虚拟账期合同”用于记录客户的永久授信额度。Id、CusFullName（客户全称）、CtClass（合同类型）、CreditAmount（信用额度）
f9_CusCredit临时授信表，记录客户临时申请的额度或账期调整，有生效和失效日期。CusBcId（关联合同ID）、TempType（类型）、TempCredit（临时额度）、EffDate/ExpDate（有效期）
f9_Customer客户基础信息表，存储客户的基本资料，如业务员（Owner）等。CusName（客户名称）、Owner（负责业务员）
f9_Company公司信息表，记录了内部公司的名称，用于排除内部公司数据。CusName（公司名称）

五、关键字段的计算规则

  • 未收金额：
    单笔未收本位币 = (原始金额 - 已收金额) × 汇率
    其中汇率根据 ChgSign 决定是乘还是除，若无汇率则视为1。
  • 客户总未收金额：
    将同一客户所有未收金额相加。
  • 客户授信额度：
    授信额度 = 虚拟合同上的CreditAmount + 有效的临时授信TempCredit
    若两者都没有（或为0），则授信额度为0。
  • 超额差额：
    超额差额 = 客户总未收金额 - 客户授信额度
    正数表示已经超额。
  • 授信状态：
    若 客户总未收金额 > 客户授信额度，则状态为“超额”，否则为“正常”。

六、业务规则说明

  • 只统计未收金额 > 0 的客户：如果客户所有收入已经收回，则无需关注，不输出。
  • 排除内部公司：内部公司的业务不参与客户信用监控，通过 NOT IN (SELECT ... FROM f9_Company) 过滤。
  • 临时授信有效期：只取当前日期在生效和失效日期之间的记录。
  • 永久授信取值：取客户 虚拟账期合同 中的 CreditAmount，若无虚拟合同则为0。
  • 业务员去重：同一个客户在客户表中可能有多条记录（如多部门），通过 MAX(Owner) 只取一个业务员，保证输出行数不膨胀。

附录：完整 SQL 代码（MySQL）

下方为完整 SQL，包含所有 CTE 和最终查询。若在 SQL Server 中使用，请将 CURDATE() 替换为 GETDATE()。

-- ============================================================ -- 客户超额授信监控 SQL -- 数据库：MySQL 兼容 -- ============================================================ WITH -- 1. 计算每笔收入明细的未收金额（本位币） CostItemDetail AS ( SELECT ci.CostId, ci.Amount, ci.ChgSign, ci.ChgRate, COALESCE(ci.Ysje, 0) AS Ysje, ROUND(COALESCE(CASE WHEN ci.ChgSign = '*' THEN (ci.Amount - COALESCE(ci.Ysje, 0)) * COALESCE(ci.ChgRate, 1) WHEN ci.ChgSign = '/' THEN (ci.Amount - COALESCE(ci.Ysje, 0)) / COALESCE(ci.ChgRate, 1) ELSE ci.Amount - COALESCE(ci.Ysje, 0) END, 0), 2) AS 单笔未收本位币 FROM f9_Costitem ci WHERE ci.Ioid = 1 ), -- 2. 获取当前有效的临时授信额度 ValidTempCredit AS ( SELECT CusBcId, TempCredit AS 临时额度 FROM f9_CusCredit WHERE TempType = 'C' AND ApproveStatus = '结束' AND CURDATE() BETWEEN EffDate AND ExpDate ), -- 3. 按客户汇总总未收金额 CustomerTotal AS ( SELECT c.Entrust AS 客户名称, SUM(ci.单笔未收本位币) AS 客户总未收本位币 FROM f9_Cost c INNER JOIN CostItemDetail ci ON c.Id = ci.CostId WHERE c.Entrust IS NOT NULL AND c.Entrust <> '' GROUP BY c.Entrust ) -- 4. 最终查询：关联授信额度、客户表，计算超额状态 SELECT ct.客户名称, cust.Owner AS 业务员, ct.客户总未收本位币, (COALESCE(vtc.临时额度, 0) + COALESCE(cc.CreditAmount, 0)) AS 客户授信额度, ROUND(ct.客户总未收本位币 - (COALESCE(vtc.临时额度, 0) + COALESCE(cc.CreditAmount, 0)), 2) AS 超额差额, CASE WHEN ct.客户总未收本位币 > (COALESCE(vtc.临时额度, 0) + COALESCE(cc.CreditAmount, 0)) THEN '超额' ELSE '正常' END AS 授信状态 FROM CustomerTotal ct -- 左关联虚拟合同（永久授信） LEFT JOIN f9_CusContract cc ON ct.客户名称 = cc.CusFullName AND cc.CtClass = '虚拟账期合同' -- 左关联临时授信 LEFT JOIN ValidTempCredit vtc ON cc.Id = vtc.CusBcId -- 左关联客户表（去重取业务员） LEFT JOIN ( SELECT CusName, MAX(Owner) AS Owner FROM f9_Customer GROUP BY CusName ) cust ON ct.客户名称 = cust.CusName WHERE ct.客户总未收本位币 > 0 AND NOT ( ct.客户名称 IN ( SELECT DISTINCT(cusname) FROM f9_Company WHERE cusname != '' ) ) ORDER BY 授信状态 DESC, ct.客户总未收本位币 DESC;

九、常见问题与注意事项

  • 为什么有些客户的授信额度为0？
    可能原因：该客户没有虚拟账期合同（永久授信为空），且当前无有效的临时授信记录。此时授信额度为0，未收金额全部视为超额。
  • 临时授信为什么可能没有关联上？
    临时授信是通过 CusBcId 关联到虚拟合同的ID。如果虚拟合同不存在，则无法关联临时授信，临时额度为0。
  • 业务员字段可能为空？
    如果客户表（f9_Customer）中没有对应的 CusName 记录，则业务员为 NULL。这表示该客户信息不完整，需维护基础数据。
  • 如何判断超额风险高低？
    超额差额越大，表示客户占用授信额度越多，风险越高。建议重点关注超额金额较大的客户。
  • 为什么排除了内部公司？
    内部公司的业务不属于外部客户信用管理范畴，因此过滤掉，只分析真正的客户。
