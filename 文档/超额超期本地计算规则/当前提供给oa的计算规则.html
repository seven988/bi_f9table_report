一、超额（Overdraft） 统一规则

1. 计算范围

根据最新 SQL，参与超额计算的费用需满足以下条件：

  • 费用表 CostItem 关联 Cost（u.CostId = c.Id 且 c.EntrustCd = u.WhoCd）
  • 关联 CusContract（虚拟账期合同）通过 c.EntrustCd = cc.CusCd 且 cc.CtClass = N'虚拟账期合同'
  • 未明确过滤 Ioid=1（实际业务中可能在其他层实现，但核心 SQL 未包含此条件，故当前规则不强制）
  • 原币比较： 实际 SQL 未显式过滤 Amount > Ysje，但通过 unrec（汇总后的未收金额）判断超额，若 unrec > 0 则可能存在未收；但为逻辑完整，仍保留原币有未收的判断，但文档按 SQL 描述。
✅ 参与计算的是： 客户应收、关联虚拟账期合同的费用（公司主体排除仍适用）。

2. 未收金额（Unpaid）—— 汇率换算及四舍五入

根据最新 SQL，金额折算采用以下规则：

  • 汇率取值： 从 ExRate 表获取最新有效汇率（EffDate 最大），目标币种 DestCurrcd = 'CNY'，且当前日期在有效期内。
  • 换算方式： 依据 ChgSign 字段：
    • 若 Currcd = 'CNY'，则折算系数为 1。
    • 若 ChgSign = '*'，则折算系数 = ChgRate。
    • 若 ChgSign = '/'，则折算系数 = 1 / ChgRate。
  • 汇总计算：
    折后费用 = Amount × 折算系数 折后已收 = Ysje × 折算系数
    分别对 折后费用 和 折后已收 做 汇总求和，最后将两个总和相减得到 unrec。
  • 精度处理： 最终结果使用 CAST(... AS DECIMAL(18,2)) 转换为保留 2 位小数的数值。
    重要 四舍五入发生在 汇总之后（即先求和，再四舍五入到分），而不是逐笔四舍五入再汇总。
📌 汇总公式：
totalAmount = CAST( SUM(Amount × 折算系数) AS DECIMAL(18,2) )
totalReceived = CAST( SUM(Ysje × 折算系数) AS DECIMAL(18,2) )
unrec = CAST( totalAmount - totalReceived AS DECIMAL(18,2) )

3. 授信额度

  • 正式授信： 从 CusContract 取 CreditAmount（取 MAX，通常只有一个）。
  • 临时授信（C类）： 从 CusCredit 汇总：
    • 条件：TempType = 'C'，ApproveStatus = N'结束'，当前日期介于 EffDate 和 ExpDate 之间。
    • 按 CusBcId（对应合同 ID）分组求和 TempCredit。
📌 可用授信 = CreditAmount + tempCredit

4. 统一授信规则（所有付款方式）

🔄 规则变更： 所有付款方式（包括现付 P、月结 M1/M2/M3/M4）不再区分，统一采用以下授信比较规则。

比较汇总未收金额 unrec（已按汇率折算并四舍五入）与可用授信 CreditAmount + tempCredit：

付款方式判定条件超额金额
所有方式
P M1 M2 M3 M4		 unrec > 可用授信 OverdraftAmount = unrec - 可用授信
若 ≤0 则无超额
✅ 现付（P）不再特殊处理，同样参与授信额度抵扣。仅当折本位币后的未收总额超过可用授信时，才判定为超额。

二、超期（Overdue） 逐笔计算（P 特殊）

超期针对 每一笔未收费用 分别计算。参与计算的费用条件与超额部分类似（均基于原币未收金额判断），但超期不涉及汇率换算，仅以业务日期和账期计算结算日。

1. 获取账期类型

根据付款方式 PayWayCd 转换为月数（仅用于非 P 方式）：

付款方式代码月数
现付P—（特殊处理）
月结M11
月结2M22
月结3M33
月结4M44
其它—0

2. 账期天数 & 临时延期（通用）

  • 合同账期天数： AccDays（如 25 天）
  • 临时延期天数（A类）： 查询 CusCredit
    • 条件：TempType='A'，ApproveStatus='结束'，当前日期有效
    • 记为 TempCredit（延期天数，若不存在则为 0）
X = AccDays + TempCredit（所有付款方式共用）

3. 结算日（SettlementDate）计算（非 P 方式）

适用于 M1/M2/M3/M4 及其他非 P 方式，沿用原有复杂逻辑。

Step 1 — 计算 Y

if X > 30: Y = X + 30 else: Y = X + 30 × payMonths

将月结月份与账期天数统一换算为“总天数”。

Step 2 — 分解为月与日

if Y 能整除 30: settleMonths = Y/30 - 1 settleDays = 30 else: settleMonths = floor(Y/30) settleDays = Y % 30

Step 3 — 基准月份

以业务日期 BizDate 取该月第一天：

基准 = YYYY-MM-01

例如 BizDate = 2026-05-18 → 2026-05-01

Step 4 — 目标月份

目标 = 基准 + settleMonths 个月

例如 2026-05-01 + 2个月 = 2026-07-01

Step 5 — 结算日

  • 若 settleDays 超过目标月份的天数（如 31 日但二月只有 28 天），则取 月末最后一天。
  • 否则：
    结算日 = 目标月份第一天 + (settleDays - 1) 天

例如：7月，settleDays=20 → 7月20日

4. 付款方式 P 的特殊处理

🔀 现付（P）超期结算日简化：
结算日 = BizDate + AccDays + TempCredit
即：业务日期 + 合同账期天数 + 临时延期天数（来自 TempType='A'）。
不再进行月份换算，直接按天累加。

示例：BizDate = 2026-05-18，AccDays = 25，TempCredit = 3 → 结算日 = 2026-06-15（5月18日 + 28天）。

5. 超期判定（所有付款方式）

超期天数 = 今天 - 结算日

SQL 实现：

DATEDIFF(DAY, SettlementDate, GETDATE())
⚠️ 若 超期天数 > 0，则 Overdue = 1（超期）；否则 Overdue = 0。
📌 最终 SQL 只返回 OverdueDays > 0 的数据，即只有已经超过结算日期的未收费用才会进入超期结果。
