基于累计阶梯算法 & 公司/个人业务分类的客服提成核算
版本 V1.0
更新 2026-06-29
数据库 SQL Server

一、概述

本 SQL 用于计算 客服人员 的销售提成。与业务员提成不同，客服提成采用 累计阶梯算法（CalcType = 'CUMULATIVE'），并且需要区分 公司业务 和 个人业务，使用不同的提成方案。核心逻辑包括：按月度汇总毛利，通过业务员名称判断业务归属类型，然后根据匹配的阶梯方案对毛利进行分段累进计算，得出最终的客服提成金额。整个计算包含 8 个逻辑层（CTE），层层递进。

三、详细逻辑说明

步骤 1：成本毛利（CTE: 成本毛利）

🎯 业务目标
与业务员提成的第一步基本一致，在 CostId 粒度汇总毛利，同时获取分单状态和主单业务大类，并额外纳入 c.service（客服人员）字段，且要求客服非空。

📥 输入数据

  • Cost：成本单头，含 ctrloffice, Saler（业务员）, service（客服）, BizDate, BusWay。
  • CostItem：收支明细及汇率信息。
  • Export：用于获取分单信息及主单业务大类。

⚙️ 核心计算逻辑

  • 毛利 = 收入合计 − 支出合计，逐行按 ChgSign 和 ChgRate 折算本币。
  • 分单识别、主单大类获取逻辑与业务员提成完全相同。

📤 输出字段
CostId, 接单公司, 业务员, 客服人员, 月度, 原始业务大类, 主单业务大类, 是否分单, 毛利金额。

⚠️ 过滤条件

  • 日期区间、未取消、业务员非空、客服非空、非红冲、业务大类非空且非'海外仓'。

步骤 2：毛利数据（CTE: 毛利数据）

🎯 业务目标
对原始业务大类进行转换（代拉代报转主单大类、跨境电商合并），然后按 接单公司, 业务大类, 业务员, 客服人员, 月度 进行聚合，为后续业务判断做准备。

📥 输入数据
来自 成本毛利。

⚙️ 核心转换规则
与业务员提成完全一致：代拉代报处理、跨境电商合并。

📤 输出字段
接单公司, 业务大类, 业务员, 客服人员, 月度, 毛利金额（已聚合）。

步骤 3：业务判断（CTE: 业务判断）

🎯 业务目标
根据业务员名称判断该笔业务属于“公司业务”还是“个人业务”，因为客服提成方案会根据此类型区分。

📥 输入数据
来自 毛利数据。

⚙️ 判断规则
若业务员满足以下任一条件，则标记为 是否公司业务 = 1，否则为 0：

  • 业务员姓名在 ('王岳祥', '孙喆') 中。
  • 业务员姓名包含 '公司'、'集团' 或 '宁波' 等关键词。

📤 输出字段
原有字段 + 是否公司业务（0/1）。

步骤 4：毛利汇总（CTE: 毛利汇总）

🎯 业务目标
将毛利按 接单公司, 业务大类, 是否公司业务, 客服人员, 月度 最终汇总，形成每个客服每月在每个业务类型下的总毛利，用于后续累计提成计算。

📥 输入数据
来自 业务判断。

⚙️ 核心逻辑
简单的 GROUP BY 求和，维度为后续方案匹配的关键。

📤 输出字段
接单公司, 业务大类, 是否公司业务, 客服人员, 月度, 毛利金额。

步骤 5：方案拆分（CTE: 方案拆分）

🎯 业务目标
从 Zj_CommissionScheme 中筛选客服提成方案（SchemeType = 'SERVICE'），且计算类型为 累计（CalcType = 'CUMULATIVE'），并拆分接单公司。

📥 输入数据
Zj_CommissionScheme 表。

⚙️ 核心逻辑

  • 使用 XML 拆分 CtrlOffice 逗号分隔的公司。
  • 过滤条件：SchemeType='SERVICE', CalcType='CUMULATIVE', Status=1, 有效期包含当前日期。
  • 注意方案表中包含 IsCompanyBusiness 字段，用于匹配公司/个人业务。

📤 输出字段
Ladder_Id, SchemeName, BusinessType, IsCompanyBusiness, 接单公司（拆分行）。

步骤 6：方案匹配（CTE: 方案匹配）

🎯 业务目标
将毛利汇总数据与拆分后的方案按 接单公司 + 业务大类 + 是否公司业务 精确匹配，获取对应的阶梯 ID。

📥 输入数据
毛利汇总 和 方案拆分。

⚙️ 核心逻辑
左连接，匹配条件为三者完全一致。

📤 输出字段
接单公司, BusinessType, IsCompanyBusiness, Ladder_Id, SchemeName。

步骤 7：提成计算明细（CTE: 提成计算明细）

🎯 业务目标
对每个客服每月的毛利，按照匹配的阶梯方案进行 累计分段计算，得到该笔毛利应计提的提成金额。该算法与业务员提成的“单笔匹配”不同，它采用 累进税制 思想：毛利金额落在不同区间，分别以不同比例计算，最终汇总。

📥 输入数据
毛利汇总、方案匹配 及阶梯表 Zj_CommissionLadder。

⚙️ 核心算法

  • 基本思路：对于每一笔毛利（可能正或负），根据阶梯区间（MinAmount ~ MaxAmount）逐段计算提成。正毛利：超出上一区间的部分乘以该区间比例；负毛利：取绝对值后同样分段，但提成金额为负（即扣减）。
  • 实现方式：使用 OUTER APPLY 将阶梯表与毛利行关联，条件为 ISNULL(l.MinAmount, 0) < ABS(毛利金额)，即所有区间左边界小于毛利绝对值的阶梯都会被关联出来，然后通过 SUM 累加每个阶梯的计算值。
  • 分段计算公式（以正毛利为例）：
    • 若毛利金额 > 当前阶梯的最大值（MaxAmount），则本段金额 = MaxAmount - MinAmount（即整个区间），乘以该阶梯比例。
    • 否则，本段金额 = 毛利金额 - MinAmount（即最后一段），乘以该阶梯比例。
    负毛利同理，但整体取负号（因为毛利为负，提成也应减少）。
  • 处理 NULL 边界：若 MinAmount 为 NULL，视为 0；若 MaxAmount 为 NULL，视为足够大（用 ABS(毛利) 本身作为上限）。
  • 最终提成：对每个阶梯的计算结果求和，得到该客服该笔毛利的提成金额（正值表示奖励，负值表示扣减）。
📌 重要说明：与业务员提成的“单笔匹配”不同，客服提成采用“累计阶梯”方式，即对毛利金额分段累进计算，而不是简单匹配一个比例。这使得提成计算更加精细，体现多劳多得。

📤 输出字段
原始毛利维度字段 + Ladder_Id, SchemeName, 客服提成金额（已汇总）。

步骤 8：最终结果（SELECT 最终）

🎯 业务目标
对计算结果进行修饰，提取匹配的具体阶梯比例、方案名称，并输出最终报表字段。

📥 输入数据
提成计算明细。

⚙️ 核心逻辑

  • 增加 规则类型：若未匹配到方案（Ladder_Id IS NULL）则为“未设置规则”，否则为“精确匹配规则”。
  • 获取 阶梯方案名称：取该阶梯下 MinAmount 最大的那一条阶梯的名称（通常表示该阶梯的最高档名称）。
  • 获取 匹配的基础提成比例：即该毛利金额落在哪个阶梯区间，取出该区间的比例（用于展示）。注意：此比例仅用于参考，实际提成金额已分段计算。
  • 格式化提成比例显示。

📤 最终输出
完整的客服提成明细报表，包含方案信息、匹配比例及最终提成金额。

五、业务规则说明

  • 客服提成方案类型：必须为 SchemeType = 'SERVICE'，且 CalcType = 'CUMULATIVE'（累计计算）。
  • 公司/个人业务划分：根据业务员姓名关键词（'公司'、'集团'、'宁波' 或特定人名）判断，分别匹配不同的阶梯方案（IsCompanyBusiness 字段）。
  • 累计阶梯算法：提成金额不是简单乘以一个比例，而是对毛利金额按阶梯分段累进计算。例如，毛利 10 万，阶梯为 0-5万(3%)，5万-10万(5%)，则提成 = 5万*3% + 5万*5%。
  • 负毛利处理：负毛利同样进行分段计算（取绝对值），但计算结果为负值，即扣减提成。
  • 未匹配方案：若未匹配到任何阶梯方案，则提成金额为 0，规则类型为“未设置规则”。
  • 数据过滤：与业务员提成一致，过滤取消、红冲、空值等，且额外要求客服字段非空。
  • 日期参数：使用 $DE_PARAM{...} 动态传入 BEGINTIME 和 ENDTIME。

SELECT c.Id AS CostId, RTRIM(LTRIM(c.ctrloffice)) AS 接单公司, c.Saler AS 业务员, c.service AS 客服人员, FORMAT(c.BizDate, 'yyyy-MM') AS 月度, COALESCE(e_main.BusWay, c.BusWay) AS 原始业务大类, MAX(CASE WHEN LIKE '%集团%' OR 业务员 LIKE '%宁波%' THEN 1 ELSE 0 END AS 是否公司业务 FROM 毛利数据 ), -- 4. 按客服、月度、业务类型汇总毛利 毛利汇总 AS ( SELECT 接单公司, 业务大类, 是否公司业务, 客服人员, 月度, SUM(毛利金额) AS 毛利金额 FROM 业务判断 GROUP BY 接单公司, 业务大类, 是否公司业务, s.CtrlOffice IS NOT NULL AND s.CtrlOffice != '' AND s.Status = 1 AND (s.EffectiveDate <= GETDATE() OR s.EffectiveDate IS NULL) AND (s.ExpirationDate >= GETDATE() OR s.ExpirationDate IS NULL) ), -- 6.
