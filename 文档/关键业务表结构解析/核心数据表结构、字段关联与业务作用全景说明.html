一、概述

本系统涉及业务员提成、客服提成、授信额度管理、客户逾期监控等多个核心模块。这些模块虽然业务场景不同，但都建立在同一套核心数据模型之上。 本文档汇总了所有 SQL 逻辑中涉及到的 8 张核心表，从业务角度解释每张表的用途、关键字段含义，以及表与表之间的关联关系，帮助开发、运维及业务人员快速建立全局数据视图。

📌 提示 文中表名可能因系统版本不同带有 f9_ 或 Zj_ 前缀，但核心字段及业务含义是一致的。

三、核心表详细说明

3.1 客户基础信息表（f9_Customer / customer）

📌 业务作用：存储所有客户（含外部客户和内部公司）的基础档案信息，是整个系统的核心主数据。

字段名业务含义关联/计算说明
Id客户唯一标识（内部ID）主键，用于系统内部关联
CusName / cusname客户简称常用于界面显示
CusFullName / cusfullnamechs客户全称关键关联字段：与合同表（CusFullName）、成本表（Entrust）进行匹配
CusCd / cuscd客户编码业务系统唯一编码
Owner负责业务员用于超额监控报表中显示业务员信息
AccDays默认账期天数当合同未指定账期时，作为兜底值

3.2 公司信息表（f9_Company / Company）

📌 业务作用：存储集团内部各分子公司信息，主要用于 过滤内部公司数据，避免将内部往来计入客户信用风险监控。

字段名业务含义关联/计算说明
CusName / shortname公司名称用于 NOT IN 子查询，排除内部公司

3.3 成本单主表（f9_Cost / Cost）

📌 业务作用：记录每笔业务（订单/委托）的成本头信息，是提成计算和未收金额统计的起点。

字段名业务含义关联/计算说明
Id成本单ID主键，与明细表（CostItem）关联
Entrust委托客户名称关键关联字段：与客户表全称匹配，用于汇总客户未收金额
Saler业务员用于销售提成归属（SchemeType='SALER'）及业务判断
Service / service客服人员用于客服提成归属（SchemeType='SERVICE'）
BizDate业务日期用于月度/季度提成周期聚合
BusWay业务大类如“跨境电商”、“代拉代报”，匹配提成方案中的 BusinessType
Cancelid取消标识='T' 表示已取消，计算时需过滤

3.4 成本明细表（f9_Costitem / CostItem）

📌 业务作用：记录成本单下的每一笔收支明细。所有 未收金额 和 毛利 的计算均基于此表。

字段名业务含义关联/计算说明
CostId关联成本单ID外键，关联 Cost.Id
Ioid收支标识1 = 收入，-1 = 支出
Amount原始金额可能为外币
Ysje已收金额（本位币）用于计算未收金额：未收 = Amount - Ysje
ChgSign汇率符号'*' 表示乘，'/' 表示除
ChgRate汇率用于将外币金额折算为本位币
redid红冲标识非 NULL 表示红冲记录，计算时需过滤（IS NULL）
WhoName对方单位用于排除内部公司（与 Company 表比对）
🔑 核心计算逻辑（未收金额）： 单笔未收本位币 = ROUND( (Amount - ISNULL(Ysje,0)) * 汇率 , 2 )
其中汇率根据 ChgSign 决定：'*' 则乘 ChgRate，'/' 则除 ChgRate，否则为 1。

3.5 合同主表（f9_CusContract / CusContract）

📌 业务作用：存储所有合同信息，包括 正式业务合同（决定永久授信额度）和 虚拟账期合同（承载临时授信与账期）。

字段名业务含义关联/计算说明
Id合同ID主键，被 CusCredit 表引用（CusBcId）
CusFullName客户全称关联客户表
CtClass合同类型'虚拟账期合同' 为虚拟合同，其他为正式合同
CreditAmount信用额度正式合同中为永久授信；虚拟合同中为基础额度
AccDays账期天数记录合同约定的账期
EffDate生效日期合同开始日期
ExpDate失效日期合同结束日期（需结合 AutoNewId 判断顺延）
CancelId取消标识='T' 表示已取消
ApproveStatus审批状态='结束' 表示已审批通过
AutoNewId自动顺延标识='T' 表示合同到期自动顺延
DeferredYears顺延年限自动顺延的年数
🔑 核心业务规则（永久授信）：
客户的 永久授信额度 = 该客户所有 有效正式合同（非虚拟、未取消、审批结束、在有效期内）中 CreditAmount 的 最大值（非总和）。
OA 调整永久额度的本质是修改某个具体正式合同的 CreditAmount。

3.6 临时授信表（f9_CusCredit / CusCredit）

📌 业务作用：存储客户的 临时额度 和 临时账期 调整记录。每个类型（TempType）同一时间只会有一条有效记录。

字段名业务含义关联/计算说明
Id记录ID主键
CusBcId关联合同ID外键，关联 CusContract.Id（通常关联虚拟合同）
TempType临时类型'C' = 授信额度调整；'A' = 临时账期调整
TempCredit临时额度/账期值当 TempType='C' 时为额度金额；TempType='A' 时为账期天数
EffDate生效日期临时调整的生效日
ExpDate失效日期临时调整的失效日
ApproveStatus审批状态='结束' 表示已审批
🔑 核心业务规则（临时额度/账期）：
系统保证 同一 CusBcId + 同一 TempType 在有效期内 只有一条记录（通过应用层或触发器控制）。
查询时直接取 GETDATE() BETWEEN EffDate AND ExpDate 的记录即可。

3.7 日志中转表（zj_ContractImportLog / zj_CreditImportLog）

📌 业务作用：作为 OA 系统与 F9 业务系统的 数据中转站。OA 审批通过后将数据写入日志表，由触发器自动同步到核心业务表（CusContract / CusCredit）。

表名用途关键字段
zj_ContractImportLog合同申请日志CtNo, CtClass, CusName, CreditAmount, EffDate, ExpDate, Status, ProcessMessage
zj_CreditImportLog临时授信申请日志CusName, TempCredit, TempType, CtNo, EffDate, ExpDate, Status, ProcessMessage

状态流转：Pending（待处理） → Success（成功） / Failed（失败）

3.8 提成配置表（Zj_CommissionScheme + Zj_CommissionLadder）

四、关键业务路径解析

以下展示核心业务场景中，数据是如何在这些表之间流转的。

4.1 提成计算路径（业务员 / 客服）

  • 数据起点：Cost（成本单） 与 CostItem（明细） 聚合出 毛利金额（收入-支出）。
  • 维度匹配：通过 Cost.Saler（业务员）或 Cost.Service（客服）以及 BusWay（业务大类），匹配 Zj_CommissionScheme 中对应的方案。
  • 比例计算：根据方案的 CalcType：
    • SINGLE（单笔）：用 ABS(毛利) 在 Zj_CommissionLadder 中匹配唯一区间，取出 CommissionRate 直接乘毛利。
    • CUMULATIVE（累计）：将 ABS(毛利) 按阶梯分段，每段分别乘以对应比例后累加。
  • 最终结果：计算出的提成金额归集到对应人员（业务员/客服）名下。

4.2 授信额度计算路径

  • 永久授信：从 CusContract 中取 CtClass <> '虚拟账期合同' 且满足有效期条件的合同，取 MAX(CreditAmount)。
  • 临时授信：从 CusCredit 中取 TempType='C' 且当前在有效期内的记录，取 TempCredit。
  • 总授信 = 永久授信 + 临时授信（若无则为 0 或兜底值 10000）。

4.3 客户未收金额与超额监控路径

  • 未收金额：从 CostItem 中取 Ioid=1（收入），排除 redid IS NOT NULL（红冲），排除 WhoName 在 Company 表中的内部公司，计算 (Amount - Ysje) * 汇率。
  • 汇总：通过 Cost 表的 Entrust 关联客户，按客户汇总未收金额。
  • 授信比较：将汇总的未收金额与上述“总授信”比较，若未收 > 授信，则标记为“超额”。
  • 合同状态过滤：通过关联 CusContract 判断客户是否有有效正式合同，用于筛选“无合同欠款”客户。

六、字段/表命名规范与版本说明

  • 前缀说明：
    • f9_：核心业务表（如 f9_Cost, f9_CusContract）
    • Zj_：系统配置/主数据表（如 Zj_CommissionScheme）
    • zj_：日志/中转表（如 zj_ContractImportLog）
  • 字段大小写：SQL Server 不区分大小写，MySQL 建议统一使用小写，但在代码中通常忽略大小写差异。
  • 关键过滤标识：
    • CancelId = 'T' 或 Cancelid <> 'T'：业务单据取消标记。
    • redid IS NULL：排除红冲明细。
    • ApproveStatus = '结束'：仅包含已审批通过的有效数据。
  • 日期函数适配：MySQL 使用 CURDATE()，SQL Server 使用 GETDATE()，业务逻辑等价。

七、总结

以上 8 张表构成了提成计算、授信控制、逾期监控三大业务模块的 数据核心。 其中，Cost 和 CostItem 是“金额计算”的源头；CusContract 和 CusCredit 是“授信额度”的载体； Zj_CommissionScheme 和 Zj_CommissionLadder 是“提成规则”的配置中心。 各表通过客户名称（CusFullName / Entrust）和合同ID（Id / CusBcId）等字段形成完整的数据链路。

理解这些表的用途和关联关系，是进行业务数据分析、SQL 优化以及问题排查的基础。建议将此文档作为团队的共享知识库使用。
